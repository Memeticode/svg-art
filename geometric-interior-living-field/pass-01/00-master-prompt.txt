## Core Product Goal

Build a full-viewport web page background made of continuously morphing SVG-based generative geometry derived from a shared visual grammar.

The result should feel like:
- a living geometric field
- always dynamic and unique
- visually coherent rather than noisy
- composed of pockets, eddies, lanes, basins, and occasional moments of local synchronization
- elegant, restrained, atmospheric, and alive

This is not:
- an icon morph gallery
- a random screensaver
- a particle toy
- a one-off animation
- a canvas/WebGL experiment

This is:
- a structured SVG generative system
- driven by deterministic code
- made from reusable modules
- suitable for use as a background layer in a real web app

## Implementation Priorities

Prioritize in this order:

1. Correct architecture
2. Deterministic generative behavior
3. Aesthetic coherence
4. Runtime performance
5. API cleanliness
6. Documentation clarity

Do not sacrifice architecture for speed unless required to make the system run at all.

## Technical Principles

You must preserve these principles throughout the implementation:

### 1. Pure state first
Geometry/state must exist as plain TypeScript data structures first.
DOM/SVG elements are only a rendering target.
Do not make the DOM the source of truth.

### 2. Deterministic generation
The system must support seeded deterministic behavior.
Given the same seed and viewport/config, it should produce materially consistent output.

### 3. Reuse a common geometric grammar
All agents/motifs must derive from a shared primitive/state grammar, not ad hoc SVG per shape.

### 4. Flow-driven behavior
Motion and morphology must be influenced by:
- flow field
- local turbulence/curl
- regional coherence
- temporal phase / inertia

### 5. Taste over noise
Randomness must always be constrained by coherent rules.
The output should look authored, not chaotic.

### 6. Pooled rendering
Avoid constant SVG node creation/destruction.
Prefer stable node pools with attribute updates.

### 7. Progressive complexity
Implement the dependency chain in a way that yields a working system early, then deepen it.

## Styling and Runtime Expectations

The visual system should aim for:
- subtle, continuous motion
- sparse but meaningful brighter events
- multi-layer depth
- low-to-medium average opacity
- occasional regional coherence
- soft palette continuity
- clear sense of drift, swirl, and local memory

Avoid:
- flashing
- hyperactive morphing
- excessive contrast everywhere
- overly literal icon readability
- unbounded randomness
- visible structural popping from frame to frame

## Visual Engine Constraints

The implementation should:
- use fullscreen SVG as the primary render surface
- render layered geometry with stable DOM nodes
- allow multiple agents across depth bands
- support morphing within a shared primitive/state schema
- support local variation without stylistic collapse

If you discover a necessary deviation from the documents, make the smallest possible deviation and document it.

## Performance Constraints

The system must be written with realistic browser performance in mind.

Minimum standards:
- avoid per-frame allocations where practical
- avoid per-frame DOM queries
- avoid destroying/recreating SVG trees during steady-state animation
- separate generation logic from render application
- scale agent counts sensibly with viewport/device constraints
- keep the architecture ready for future optimization
